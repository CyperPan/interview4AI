## Continuous Batching

### 什么是 Continuous Batching（持续批处理）？为什么它对 LLM 推理很重要？

**答：**

**痛点：**

传统 Static Batching 必须等 Batch 中最长的那个句子生成完毕，整个 Batch 才能退出。这导致提前生成完的短句子对应的 GPU 算力处于"空转（Bubble）"状态，资源浪费严重。

```
Static Batching:
时间 →
请求1: [==========]                    ← 短请求，已结束但占用资源
请求2: [===============================]  ← 长请求，决定 batch 结束时间
请求3: [======]                        ← 短请求，已结束但占用资源
           ↑
        资源浪费区
```

**原理：**

Continuous Batching（也叫 In-flight Batching）打破了请求级的边界，实现了 **Token 级调度**。

```
Continuous Batching:
时间 →
Iter1: [请求1][请求2][请求3]
Iter2: [请求1][请求2][请求3]
Iter3: [请求2][请求4]      ← 请求3 结束，请求4 进来
Iter4: [请求2][请求4][请求5] ← 请求1 结束，请求5 进来
```

当 Batch 中某个请求遇到 `<EOS>`（生成结束）时，调度器会立刻将它踢出，并从等待队列中塞入一个新的请求参与下一次 Token 的生成。

**重要性：**

它将 GPU 的吞吐量（Throughput）提升了数倍，是目前 vLLM、TensorRT-LLM 等所有现代推理引擎的标配。

## PagedAttention

### 什么是 PagedAttention？它解决了什么问题？

**答：**

**痛点：**

LLM 解码时 KV Cache 会随着序列增长而动态增大。

传统框架（如早期 FasterTransformer）会按最大可能长度（如 2048）预先分配连续显存。这会导致：
- **内部碎片**：句子没那么长，预分配的空间浪费
- **外部碎片**：显存不连续无法分配
- **结果**：超出一半的显存被浪费，严重限制了并发量（Batch Size）

```
传统分配：
显存: [████    ] [████    ] [████    ]  ← 每个请求预分配 2048
      请求1(512)  请求2(1024) 请求3(512)
      浪费:75%    浪费:50%    浪费:75%

外部碎片：
显存: [██][  ][██][  ][██]  ← 空闲但不连续，无法分配给大请求
```

**解决方案：**

借鉴操作系统的**虚拟内存分页机制**：

```
PagedAttention:
┌─────────────────────────────────────┐
│           逻辑视角                   │
│  请求1: [Block0][Block1][Block2]     │
│  请求2: [Block3][Block4]             │
└─────────────────────────────────────┘
            ↓ BlockTable 映射
┌─────────────────────────────────────┐
│           物理视角                   │
│  显存: [B0][B3][B1][B4][B2][空闲]    │
│       离散存储，无外部碎片            │
└─────────────────────────────────────┘
```

- 将连续的 KV Cache 划分为固定大小的 Block（如每个 Block 存 16 个 Token）
- 在逻辑上请求看到的是连续的 Token
- 在物理 HBM 上，它们被**离散地存储**
- 通过 **BlockTable** 记录逻辑到物理的映射

**收益：**

- 彻底消除外部碎片
- 内部碎片限制在最后一个 Block 内（< 4% 浪费）
- 让同样的 GPU 能扛住 **3-4 倍**的并发请求

## PD 分离

### 为什么很多高性能推理系统会考虑做 Prefill-Decode Disaggregation（PD 分离）？

**答：**

**根本原因（算力特征冲突）：**

| 阶段 | 计算特征 | 瓶颈类型 |
|-----|---------|---------|
| **Prefill** | 一次性处理长 Prompt，GEMM | **Compute-bound**（算力密集） |
| **Decode** | 每次只吐一个 Token，GEMV | **Memory-bound**（访存密集） |

**混部的问题：**

如果把这两者放在同一个 GPU 上处理，当一个新的长请求进来做 Prefill 时，会瞬间抢占算力，导致正在做 Decode 的请求被"卡住"，引起严重的**尾延迟（Tail Latency 抖动）**。

```
混部时间线：
时间:  [====][====][========][====][====]
       D1    D2    P1(新请求) D1    D2
                   ↑
            D1, D2 被卡住！

PD 分离：
Prefill 节点: [========][========]
              P1        P2
                   ↓ RDMA 传输 KV
Decode 节点:  [====][====][====][====]
              D1    D2    D3    D4
              无干扰，延迟稳定
```

**PD 分离方案：**

将 Prefill 和 Decode 拆分到不同的 GPU 甚至不同机器上（如 Kimi 采用的 Mooncake 架构，或开源的 DistServe）：
1. Prefill 节点专门算 Prompt
2. 算完后把 KV Cache 通过网络（RDMA）传给 Decode 节点
3. 两边都能针对性调优，极大保障了线上 **P99 延迟的稳定性**

## 投机解码

### 什么情况下投机解码（Speculative Decoding）能够带来明显收益？

**答：**

**核心逻辑：**

投机解码是用一个极小的 Draft 模型（如 1B）快速猜出几个 Token，再用大模型（Target，如 70B）一次性验证。

```
流程：
Step 1: Draft 模型快速生成 K 个候选 Token
        [Token1][Token2][Token3][Token4]
        
Step 2: Target 模型并行验证（一次前向）
        [✓][✓][✓][✗]
        
Step 3: 接受前 3 个，从第 4 个重新生成
```

**它能加速的本质：**
- 利用大模型 Decode 阶段**过剩的算力**（空闲的 SM 单元）
- 利用富余的显存带宽
- 把串行的 Token 生成变成并行的验证

**适用场景：**

| 场景 | 说明 | 原因 |
|-----|------|------|
| **高重复度输出** | 代码补全、生成固定格式的审核 JSON | 小模型命中率（Acceptance Rate）极高 |
| **模板化内容** | 输出有强模式可循 | Draft 容易猜对 |

**不适用场景（反套路）：**

- **Low Batch Size（低并发）** 才适合
- 如果系统已经满载（高 QPS），大模型的算力已经被榨干
- 此时引入小模型猜词不仅不能加速，反而会抢占宝贵的显存带宽，导致整体吞吐下降

## 量化策略

### 量化为什么能提速？INT4 和 FP8 你怎么选？为什么有些量化方案精度掉得很厉害？

**答：**

**为什么提速：**

LLM 推理（尤其是 Decode 阶段）是典型的 **Memory-bound（访存瓶颈）**，算力大量空转等待权重从 HBM 搬运到 SRAM。

量化（如把 FP16 变 INT8/INT4）直接把数据体积缩减了 2 倍到 4 倍，**极大降低了访存延迟**，这是提速的根本原因。

**FP8 vs INT4 怎么选：**

| 量化类型 | 适用场景 | 特点 |
|---------|---------|------|
| **FP8** | Hopper 架构（H100）首选 | 硬件原生支持，不需要反量化开销；保留指数位，动态范围大，对大模型精度保护极其优秀 |
| **INT4** | 极致压榨单卡并发量（如 4090 跑 70B） | Weight-Only 量化（AWQ/GPTQ）；计算前需要解包（Unpack）回 FP16，有一定计算开销 |

**为什么掉点厉害（杀手锏）：**

因为 LLM 当参数量超过 6.7B 之后，激活值（Activation）中会出现**极大量的 Outliers（异常极大值）**。

如果用朴素的 PTQ（训练后量化），这些异常值会被直接截断，导致精度断崖式下跌。

**解决方案：**
- **SmoothQuant**：把激活的难度转移到权重上
- **AWQ**：保留那 1% 最重要的显著权重不量化

## 算子融合

### 算子融合（Kernel Fusion）为什么重要？

**答：**

**核心原因：**

在推理中，由于 Decode 阶段是 Memory-bound，限制速度的不是 GPU 的浮点运算能力，而是去显存（HBM）中读写数据的速度。

**问题：**

如果按照 PyTorch 的原生写法，一个公式包含乘法 -> 加法 -> 激活函数，这需要调用 3 个 CUDA Kernel：
1. 每次启动 Kernel 都有百纳秒级的 Launch 开销
2. 前一个 Kernel 计算的中间结果要写回 HBM
3. 后一个 Kernel 又要从 HBM 读出来

**算子融合解决方案：**

把这些步骤写成一个定制的 CUDA C 算子：
```cuda
// 融合前：3 个 Kernel，多次 HBM 读写
// Kernel 1: tmp = x * w
// Kernel 2: tmp = tmp + b  
// Kernel 3: out = activation(tmp)

// 融合后：1 个 Kernel，数据在 SRAM 中传递
// 所有中间数据都在 GPU 芯片内部速度极快的 SRAM（Shared Memory）和寄存器中完成传递
// 绝不写回主存
```

**收益：**

极大降低访存延迟，是推理底层加速的绝对核心。如 FlashAttention、融合 RMSNorm 都是典型案例。
